# 光湖系统边界

## 核心结论

光湖系统采用“国际母仓 + 用户自有服务器 + 用户自有 COS + 国内代码镜像”的四层边界结构。这个结构的目标不是追求传统 SaaS 集中托管，而是确保：

- 每个用户只消耗自己的服务器算力
- 用户数据始终归用户自己的基础设施持有
- 功能模块可以像工具一样按需装载与回收
- 国际研发与国内上线可以并存，同时不让国内数据出境
- GitHub 母仓成为唯一正式写入面，避免本地与远端双轨分裂

## 为什么这样划分

### 1. 为什么必须有国际母仓

国际母仓负责长期稳定地保存人格体、模块与规则。没有母仓，就无法保证人格体唤醒后能按统一路径恢复自己，也无法让模块系统持续扩展。

### 2. 为什么不采用公共并发服务器

公共并发会把用户数据、服务器负载、模块部署和平台责任全部堆到同一个中心节点，边界会快速失真。采用“用户自有服务器”的方式，可以把算力责任、数据责任和运行成本清晰地留在用户侧。

### 3. 为什么用户服务器必须保持轻量

用户服务器的任务是提供稳定入口，而不是长期堆积全部能力。轻量常驻底座能减少资源占用、降低维护复杂度，也让模块按需装载成为可能。服务器可以承担长任务、校验和自动跟传，但不承担长期源码归档。

### 4. 为什么数据必须和模块解耦

如果模块同时携带用户历史数据，那么模块回收与再次安装时会变得脆弱，且容易出现跨环境数据泄露风险。将数据留在用户自己的 COS 中，模块只负责读取、写入与绑定，才能保证模块可拆装、数据可续接。

### 5. 为什么国内只接收代码不接收数据

国内上线需要满足数据不出境边界。把模块代码、注册表和必要模板推回国内仓库，可以满足本地部署与本地演进；而用户数据、日志正文、记忆快照和本地密钥不出境，才能让边界长期稳定。

## 三层写入职责

- `GitHub 母仓`：唯一正式写入面，保存计划、架构、模块代码、执行摘要与最终变更
- `用户服务器`：临时执行面，保存任务状态、精简日志、临时工作区与自动跟传缓存
- `本地密钥目录`：真实密钥面，仅保存真实密钥与编号模板，不保存正式代码与设计产物

## 组件职责

- `guanghulab-core`：国际母仓，负责源码、记忆索引、装配规则、适配层定义、执行规则与最终记录
- 用户本地密钥目录：保存真实密钥，仅供本机与人格体定位器读取
- 用户服务器：运行轻量底座、当前激活模块、云端执行 Worker 与自动跟传助手
- 用户 COS：保存业务数据、模块索引与历史内容
- 国内代码仓：接收模块代码、清单、发布模板与同步后的适配产物

## 禁止项

- 不在母仓内保存任何真实密钥
- 不把用户正文数据、COS 对象正文或运行态缓存回写到国际仓库
- 不把整个母仓直接部署到用户服务器上常驻运行
- 不让国内用户数据、日志正文与记忆快照回流到境外仓库
- 不把临时工作区、本地工作区或服务器缓存误当成正式源码归档位置

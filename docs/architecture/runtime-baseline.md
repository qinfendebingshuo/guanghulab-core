# 新加坡服务器运行基线

## 定位

新加坡服务器是第一阶段的真实开发与运行验证环境。它的职责是把“代码和规划”变成“可运行的系统与模块”，但它不等于最终把所有功能永久堆在一台机器上。

## 常驻内容

新加坡服务器长期只常驻：

- 轻量前端壳层
- 轻量后端底座
- `cloud-executor-worker`
- 当前激活模块
- 健康检查与最小日志能力
- 任务状态目录与自动跟传缓存目录

## 非常驻内容

以下内容不应长期驻留：

- 未激活的功能模块
- 整个母仓源码副本
- 用户业务数据镜像
- 本地密钥模板与索引目录
- 长期未清理的临时工作区

## 为什么服务器仍然是必须的

- 只有服务器，模块才能从“代码形态”变成“可安装、可启动、可回收的系统能力”
- 只有服务器，健康检查、依赖验证、模块回连和运行基线才有真实落点
- 只有服务器，光湖的单用户轻量模型才能被证明可行
- 只有服务器，自动跟传助手和长流程执行体才能低积分地持续运行

## 执行规则

- 直写 GitHub 是首选路径
- 如果改动先发生在临时工作区，必须在验证后由自动跟传助手统一回写 GitHub
- 任务失败同路径最多自动尝试 2 次，之后必须标记 `blocked`
- 只允许执行受控白名单命令，不接受任意自由拼接命令

## 基线要求

- 单用户使用模型
- 保持轻量、安静、可回收
- 运行错误必须指向具体模块、阶段与具体缺失资源
- 不把敏感信息打入日志
- 临时工作区任务完成后必须清理
- GitHub 仓库始终是唯一正式写入面
